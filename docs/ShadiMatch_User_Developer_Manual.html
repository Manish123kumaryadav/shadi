1. Project Overview

ShadiMatch is a matrimony-style matching application. It allows users to register, verify login using mobile OTP, browse compatible profiles, like or pass profiles, start conversations, and receive offline message email notifications.

The application is split into two parts:

  • Frontend: React app inside the src folder.
  • Backend: Express API and Socket.IO server inside the server folder.

Folder Structure

PathPurpose
src/pagesReact pages such as Home, Register, Login, Browse, Messages, Profile, Likes, and footer information pages.
src/componentsReusable UI components such as Navbar and ProfileCard.
src/services/api.jsAxios API client and Socket.IO client setup.
server/routesExpress routes for auth, profiles, matches, and messages.
server/modelsSequelize models and relationships.
server/socketSocket.IO connection and room handling.
server/servicesPresence and email notification services.

2. Main Features

  • Mobile number collected during registration.
  • OTP based login using registered mobile number.
  • JWT authentication for protected API routes.
  • 10 male and 10 female seeded test users.
  • Male test user sees 10 female matches.
  • Female test user sees 10 male matches.
  • Like, pass, views, and conversation start support.
  • Realtime chat using Socket.IO rooms.
  • Email notification when recipient is offline.
  • Interactive footer pages for About, Blog, Careers, Help, Contact, FAQ, Privacy, Terms, and Safety.

4. Setup and Run

Install Dependencies

npm install

Configure Environment

Edit .env in the project root:

PORT=5000
CLIENT_URL=http://127.0.0.1:5173,http://localhost:5173

DB_HOST=127.0.0.1
DB_PORT=3306
DB_NAME=shadiDB
DB_USER=root
DB_PASSWORD=

JWT_SECRET=change-this-secret-before-production

Start Backend

npm run server

Backend URL:

http://127.0.0.1:5000

Start Frontend

npm run dev

Frontend URL:

http://127.0.0.1:5173

Build Frontend

npm run build

5. Database and Tables

The backend uses Sequelize ORM. On startup, the server runs:

sequelize.sync({ alter: true })

This creates or updates the required MySQL tables automatically.

Main Tables

TablePurpose
UsersStores account, mobile, OTP, gender, lookingFor, DOB, and auth data.
ProfilesStores religion, caste, location, education, occupation, height, bio, and interests.
PhotosStores profile photo URLs.
LikesStores like/pass actions.
ProfileViewsStores profile view records.
ConversationsStores chat conversation records.
ConversationMembersMaps users to conversations.
MessagesStores chat messages.

6. Registration and OTP Login

Registration Flow

  1. User opens /register.
  2. User enters full name, email, mobile number, password, DOB, religion, and preferences.
  3. Backend creates the user and profile.
  4. User is redirected to /login.

OTP Login Flow

  1. User opens /login.
  2. User enters registered mobile number.
  3. Backend generates a 6 digit OTP.
  4. User enters OTP.
  5. Backend verifies OTP and returns JWT token.
  6. Frontend stores token in localStorage and redirects to /browse.
Development note: OTP is currently returned as devOtp and shown in the UI. For production, connect an SMS provider and remove OTP display from the frontend.

8. Matching Rules

The match API returns compatible profiles based on gender preference:

  • If logged in user is male and looking for female, API returns female profiles looking for male.
  • If logged in user is female and looking for male, API returns male profiles looking for female.
  • By default, the API returns 10 matches for testing.
  • Filters can apply age, location, and religion.

9. Realtime Chat and Offline Email

Realtime Chat

  1. User opens a matched profile and clicks message.
  2. Frontend calls POST /api/messages/start/:profileId.
  3. Backend creates or reuses a conversation.
  4. Messages page joins the Socket.IO conversation room.
  5. New messages are broadcast with message:new.

Offline Email Notification

If the recipient is not connected to Socket.IO, the backend sends an email notification with the message content.

SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_SECURE=false
SMTP_USER=your_email@gmail.com
SMTP_PASS=your_app_password
EMAIL_FROM=your_email@gmail.com
If SMTP settings are empty, the backend logs the email notification in the server console instead of sending it.

11. API Reference

Auth APIs

MethodPathPurpose
POST/api/auth/registerCreate account and profile.
POST/api/auth/send-otpGenerate OTP for registered mobile.
POST/api/auth/verify-otpVerify OTP and return JWT.
POST/api/auth/loginLegacy email/password login for API testing.
GET/api/auth/meReturn logged in user and profile.

Match APIs

MethodPathPurpose
GET/api/matchesList compatible matches.
POST/api/matches/:profileId/likeLike a profile.
POST/api/matches/:profileId/passPass a profile.
GET/api/matches/likesList received likes.
GET/api/matches/viewsList profile viewers.

Message APIs

MethodPathPurpose
POST/api/messages/start/:profileIdCreate or open conversation.
GET/api/messages/conversationsList conversations.
GET/api/messages/conversations/:conversationIdList messages.
POST/api/messages/conversations/:conversationIdSend message.

12. Troubleshooting

CORS Error

  • Ensure backend is running on http://127.0.0.1:5000.
  • Ensure frontend origin is listed in CLIENT_URL.
  • Restart backend after changing .env.

Database Connection Fails

  • Confirm MySQL is running.
  • Confirm database shadiDB exists.
  • Check DB_USER and DB_PASSWORD in .env.

OTP Not Working

  • Use a registered mobile number.
  • Check Development OTP shown on login page.
  • OTP expires after 5 minutes.

No Matches Showing

  • Restart backend so seed users are inserted.
  • Log in with one of the seeded test mobile numbers.
  • Clear filters in Browse page.

13. Production Checklist

  • Change JWT_SECRET to a strong private value.
  • Disable returning devOtp from /api/auth/send-otp.
  • Connect a real SMS provider for OTP delivery.
  • Configure SMTP credentials for offline email notification.
  • Use HTTPS for frontend and backend.
  • Move secrets out of committed files.
  • Add rate limiting for OTP and login APIs.
  • Add stronger validation for profile data and uploaded images.
  • Replace demo image URLs and seed data with production content.
  • Use migrations instead of sequelize.sync({ alter: true }) for production database changes.
